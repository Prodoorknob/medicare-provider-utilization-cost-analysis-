# AllowanceMap — Areas for Improvement

> Last updated: 2026-04-08
> Current state: Phase 6 complete (pipeline + models + web app deployed)

---

## 1. Synthetic Data

### Current State

Stage 2 (out-of-pocket prediction) relies entirely on synthetic beneficiary data generated by `generate_synthetic_mcbs.py`. The synthetic pipeline:

- Loads real MCBS Cost Supplement distributions for age, sex, income, and chronic condition counts
- Samples demographics from these year-level distributions
- Joins with real provider gold data (allowed amounts, risk scores, specialty, etc.)
- Modulates OOP amounts using hardcoded factors:
  - Dual eligible (Medicare + Medicaid): -85% reduction
  - Supplemental insurance: -35% reduction
  - Chronic condition count: +3% per condition above baseline
  - Log-normal noise applied for variance
- Assigns Census region based on state-to-region mapping

### Problems

**Hardcoded modulation factors are not learned from data.** The -85% dual eligible reduction and -35% supplemental reduction are rough approximations from literature, not estimated from actual claims. The real relationship between dual status and OOP is more nuanced — it varies by service type, state Medicaid rules, and specific plan details.

**No real beneficiary-level linkage.** The MCBS PUF has two independent samples (Survey File and Cost Supplement) with non-overlapping PUF_IDs. This means we cannot observe the same individual's demographics AND their OOP spending. The synthetic data fabricates this linkage.

**Census region assignment is deterministic.** Every provider in California is assigned region 4 (West). In reality, beneficiaries may receive care across state lines, and regional cost variation within a Census region (e.g., rural Montana vs. urban California, both "West") is substantial.

**No plan-level detail.** Real OOP depends on specific Medigap plan type (A through N), Part D plan deductibles, Medicare Advantage copay structures, and state-specific Medicaid benefits. The synthetic data has only binary flags (dual yes/no, supplemental yes/no).

**OOP distribution shape is assumed.** The synthetic generation uses log-normal noise, but real OOP distributions are often zero-inflated (many services have $0 OOP for dual-eligible beneficiaries) with heavy right tails for catastrophic costs.

### Improvement Paths

- **MCBS Limited Data Set (LDS):** $600 per module, requires Data Use Agreement from CMS. Provides real per-beneficiary OOP spending linked to demographics including Census region. The existing pipeline (`08_mcbs_crosswalk_local.py --mode synthetic` → `--mode puf`) is designed for drop-in replacement.
- **CMS Virtual Research Data Center (VRDC):** 100% Medicare claims at beneficiary level. Requires institutional researcher access and approved DUA. The gold standard but hardest to obtain.
- **Truven/IBM MarketScan:** Commercial + Medicare claims with actual plan-level detail (deductibles, copays, plan type). Available through university data license agreements.
- **Intermediate improvement:** Instead of hardcoded factors, fit a simple regression on the real MCBS PUF Cost Supplement data to learn the OOP modulation as a function of demographics. Even at the national aggregate level, this would produce data-driven coefficients rather than literature estimates.

## 2. Models

### GLM (SGD) — Completely Diverged

The GLM achieved R² = -102.80, meaning it predicts worse than simply guessing the mean. Root causes:

- **No feature scaling.** SGDRegressor is sensitive to feature magnitudes. `Avg_Sbmtd_Chrg` ranges from 0 to millions, while `place_of_srvc_flag` is 0/1. The gradient steps are dominated by the largest-magnitude feature.
- **Huber loss with default epsilon.** The epsilon parameter controls the threshold between squared and linear loss. The default may not suit the Medicare cost distribution.
- **Learning rate schedule.** The current `adaptive` schedule may be stepping too aggressively or not converging.
- **No regularization tuning.** The `alpha` parameter controls L2 penalty strength and was not searched.

**Fix:** Replace with a proper pipeline: `StandardScaler` → `ElasticNet` (or `Ridge`) with cross-validated alpha. Alternatively, use `SGDRegressor` with `StandardScaler` in a `Pipeline` and grid-search `alpha`, `epsilon`, and `learning_rate`.

### XGBoost Underperforms Random Forest

XGBoost (R² = 0.833) underperforms RF (R² = 0.884), which is unusual for tabular data. Likely causes:

- **Batch training artifact.** The default `--mode batch` trains incrementally by Census region (125 rounds per region, passing the booster forward via `xgb_model` param). The ordering of regions matters — if the first region biases the tree structure, later regions may not recover. The booster is essentially fine-tuned 5 times sequentially.
- **Insufficient rounds per region.** 125 rounds may not be enough for each regional slice. The total is 625 rounds but each sees only a fraction of the data.
- **No hyperparameter search.** RF uses `RandomizedSearchCV` in full mode; XGBoost uses fixed parameters (lr=0.05, max_depth=6, subsample=0.8).
- **Early stopping behavior in full mode.** The `--mode full` with early stopping on a single validation set may stop too early if the validation set is not representative.

**Fix:** Run `--mode full` with `RandomizedSearchCV` or Optuna, searching over `learning_rate` (0.01-0.3), `max_depth` (4-10), `min_child_weight` (1-10), `subsample` (0.6-1.0), `colsample_bytree` (0.6-1.0), `gamma` (0-5). This should close or exceed the RF gap.

### LSTM Metrics Not Comparable

The LSTM R² = 0.886 is computed on group-level temporal predictions (mean allowed amount per specialty × bucket × state for years 2022-2023). This is predicting ~17K group-year means, not individual service-level records like RF/XGB which predict on millions of individual rows.

**The comparison is misleading.** Group-level means are smoother and easier to predict. The LSTM would likely have a lower R² if evaluated on the same individual-record test set.

**Fix:** Add an apples-to-apples comparison. Either:
1. Evaluate RF/XGB at the group-year level (aggregate their predictions, then compute R²)
2. Use the LSTM group-level predictions as a feature in the RF/XGB models (hybrid approach)
3. Clearly separate "individual prediction" and "trend forecasting" as different tasks in documentation

### No Ensemble Methods

The current pipeline trains 4 independent models and compares them. No model combination is attempted.

**Potential approaches:**
- **Stacking:** Use RF and XGBoost predictions as features in a meta-learner (Ridge or another XGBoost). Typically adds 1-3% R² on tabular tasks.
- **Weighted average:** Simple weighted blend (e.g., 0.6 × RF + 0.4 × XGB) based on validation performance.
- **Temporal ensemble:** Use LSTM for the temporal component (year-over-year trend) and RF/XGB for the cross-sectional component (within-year variation). Combine predictions additively.

### OOP Quantile Model

The Stage 2 OOP quantile regression achieves P50 R² = 0.40. While this is partly due to synthetic training data, the model architecture could also be improved:

- **No feature interactions.** XGBoost learns some interactions, but explicitly constructing `allowed_amt × dual_eligible` or `chronic_count × age` could help.
- **Quantile crossing.** Three independent boosters (P10, P50, P90) can produce crossing quantiles (P10 > P50). A joint quantile model or post-hoc sorting would prevent this.
- **No monotonicity constraints.** We know dual eligibility should decrease OOP, higher chronic count should increase OOP. XGBoost `monotone_constraints` parameter can enforce this.

## 3. Training

### No Cross-Validation

All models use a single 80/20 random split with `random_state=42`. This means:

- The reported metrics depend on which specific 20% was held out
- There's no estimate of metric variance (is MAE = $12.04 ± $0.50 or ± $3.00?)
- Overfitting to the specific test set is possible, especially after multiple rounds of model development

**Fix:** Implement 5-fold cross-validation. For the full 103M dataset, stratified group k-fold (stratified by state or specialty) would ensure each fold has representative coverage. Report mean ± std for all metrics.

### Batch Training Region Ordering Bias

XGBoost and RF batch modes train sequentially over Census regions: Northeast → South → Midwest → West → Territories. For XGBoost, the booster is passed forward between regions via `xgb_model`. For RF, `warm_start` adds 125 trees per region.

**Problems:**
- The first region (Northeast) shapes the initial tree structure for XGBoost; later regions fine-tune but may not fully correct initial biases.
- The order is deterministic — shuffling regions would give different results.
- Territories (smallest region, includes PR, GU, VI, AS) trains last with the fewest data points on top of a booster already fit to 50 states.

**Fix:** Either use full mode (train on all data at once) or randomize region order across multiple runs and ensemble the results.

### LSTM Potential Underfitting

The LSTM early-stopped at epoch 19 of 100, suggesting the learning rate schedule reduced too aggressively or patience was too low.

- **Validation loss trajectory:** train_loss was still decreasing when validation loss plateaued. This could indicate the model capacity is sufficient but the learning rate dropped too fast.
- **Architecture search:** Only one configuration was tried (hidden=64, 2 layers, embed=8, dropout=0.2). The model has 213K parameters for 20K training groups, which may be underpowered.
- **No learning rate warmup.** Starting at lr=1e-3 and immediately training can cause instability in the first few epochs.

**Fix:**
1. Try larger models: hidden=128 or 256, embed_dim=16, 3 layers
2. Increase patience from 10 to 20
3. Add learning rate warmup (linear warmup over 5 epochs)
4. Use cosine annealing scheduler instead of ReduceLROnPlateau
5. Run Optuna hyperparameter search over hidden_size, num_layers, embed_dim, dropout, lr

### Sample Fraction

RF and XGB default to `--sample 0.3` (30% of data) for memory management. This means models only see ~31M of the 103M available records.

**Impact:** Random Forest in particular benefits from more data — each tree's bagging sample is 30% of 30%, meaning only ~9% of total records influence any individual tree.

**Fix:** With the 5070 Ti's 16GB VRAM, full-data XGBoost training should be possible using the `hist` tree method with `max_bin=256`. RF full mode may need cuML (GPU) or a machine with 64GB+ RAM.

### No Temporal Validation for Tree Models

RF and XGB use random 80/20 splits, which means training data includes 2023 records and test data includes 2013 records. This is temporal leakage for any year-dependent patterns.

**Fix:** Use temporal splits for tree models too: train on 2013-2021, test on 2022-2023. This gives a fair comparison with the LSTM and tests whether models generalize to future years.

## 4. Feature Engineering

### Submitted Charge Dominance

`Avg_Sbmtd_Chrg` has feature importance of 0.618 in the Random Forest — it explains nearly two-thirds of the prediction on its own. This is because submitted charges and allowed amounts are mechanically related: Medicare applies fee schedule reductions to the submitted charge.

**Why this is a problem:**
- The model is essentially learning the fee schedule reduction ratio, not the underlying cost drivers
- In a real prediction scenario, you might not know the submitted charge before the service is rendered
- It inflates R² without demonstrating that the model learned meaningful patterns

**Fix:**
1. Train a model variant that excludes `Avg_Sbmtd_Chrg` entirely. Compare R² drop — this is the "honest" model performance.
2. If charge is retained, clearly document it as a "charge-informed" model vs. a "charge-free" model.
3. Replace submitted charge with the Medicare Fee Schedule RVU values (from MPFS), which are publicly known before the service. This converts the model from "predict allowed from charge" to "predict deviation from expected fee schedule."

### No Interaction Features

The current feature set treats all 10 features as independent. Known interactions in Medicare data:

- **Specialty × State:** Some specialties have very different practice patterns by state (e.g., orthopedic surgery costs vary 2-3x between states due to different implant usage)
- **Specialty × Bucket:** An Internal Medicine provider doing radiology services has a different cost profile than a Radiologist doing radiology
- **POS × Bucket:** Facility vs. office differential varies dramatically by service type (surgery has large facility fee, E&M has small)
- **Risk Score × Specialty:** High-risk patients seeing specialists generate different costs than high-risk patients in primary care

**Fix:** Create interaction features: `specialty_bucket = specialty_idx * 6 + bucket`, `pos_bucket = pos * 6 + bucket`. Or use target encoding for these combinations. Tree models can learn interactions, but explicit features help with shallower trees and faster convergence.

### HCPCS Code Encoding

`HCPCS_Cd_idx` is a raw label-encoded integer for ~6,000 unique codes. Problems:

- **Arbitrary ordering.** Code index 100 is not "between" codes 99 and 101 in any meaningful way. Tree models handle this but waste splits finding the right thresholds.
- **High cardinality.** 6,000 categories dilute signal in Random Forest — each split on HCPCS_Cd has thousands of possible thresholds.
- **No semantic structure.** CPT codes have a natural hierarchy (e.g., 99213 and 99214 are adjacent E&M levels) that label encoding destroys.

**Fix:**
1. **Target encoding:** Replace HCPCS_Cd_idx with the historical mean allowed amount for that code. This creates a continuous feature with direct predictive power.
2. **Frequency encoding:** Replace with log(count of records for that code). Common codes may have different cost dynamics than rare ones.
3. **Hierarchical encoding:** Use the first 2-3 digits of the CPT code as a mid-level category (finer than `hcpcs_bucket` but coarser than full code). This creates ~200 groups instead of 6,000.
4. **Embeddings (for LSTM):** Already done for the LSTM; could extract learned embeddings and use as features for tree models.

### Missing Temporal Features for Tree Models

The `year` column is preserved through the pipeline but explicitly not used as a feature for RF/XGB/GLM. This discards useful temporal information:

- **Medical inflation:** Allowed amounts increase ~2-4% annually. Without year as a feature, the model cannot distinguish a 2013 record from a 2023 record.
- **Policy changes:** Medicare fee schedule updates, sequestration adjustments, and COVID-era waivers all create year-specific patterns.

**Fix:**
1. Add `year` as a feature for tree models (it's already in the gold schema)
2. Create derived features: `years_since_2013`, `is_covid_era` (2020-2021), `inflation_factor` (cumulative Medical CPI since 2013)
3. Add year-over-year change features: for each group, the rate of change from the previous year

### Risk Score Imputation

Missing `Bene_Avg_Risk_Scre` values are imputed with the global median (~1.0). This is a rough approach because:

- Risk scores vary dramatically by specialty (oncology has higher average risk than optometry)
- The proportion of missing risk scores may vary by year and state
- Imputing with a constant creates an artificial spike in the risk score distribution at 1.0

**Fix:**
1. Use specialty-specific median imputation
2. Use a secondary model (predict risk score from specialty, state, and service volume)
3. Add a binary `risk_score_missing` indicator feature so models can distinguish real 1.0 from imputed 1.0

### Untapped Raw Data Fields

The CMS "by Provider & Service" dataset contains additional columns not currently extracted:

- **`Rndrng_Prvdr_Gndr`** — Provider gender (may correlate with specialty mix and cost patterns)
- **`Rndrng_Prvdr_Ent_Cd`** — Entity code (individual vs. organization)
- **`Rndrng_Prvdr_Cntry`** — Country (for filtering non-US providers)
- **`HCPCS_Drug_Ind`** — Drug indicator (flags Part B drug codes, which have very different cost dynamics)
- **`Tot_Drug_Srvcs` / `Tot_Drug_Benes`** — Drug-specific service counts

## 5. Data Pipeline

### Bronze Ingest Memory Issues

`01_bronze_ingest_local.py` OOMs when processing all 6,313 partitioned files. The current workaround is skipping bronze and reading directly from partitioned_data in the silver stage.

**Root cause:** The script tries to load all files into a single DataFrame before writing. With ~10M rows per year × 11 years, the in-memory representation exceeds 32GB.

**Fix:**
1. Stream-process files in chunks: read one file, append to parquet, release memory
2. Use PyArrow's `ParquetWriter` to write incrementally without holding all data in memory
3. Use Dask or Polars for lazy evaluation with spill-to-disk
4. Accept that bronze is a Databricks/Spark job and skip it locally (the silver stage already handles the raw data fine)

### Aggressive Outlier Removal

The silver stage applies 3× IQR outlier removal on `Avg_Mdcr_Alowd_Amt`:

```python
Q1 = df[col].quantile(0.25)
Q3 = df[col].quantile(0.75)
IQR = Q3 - Q1
lower = Q1 - 3 * IQR
upper = Q3 + 3 * IQR
df = df[(df[col] >= lower) & (df[col] <= upper)]
```

**Problems:**
- Medicare costs are inherently right-skewed. High-cost surgical procedures, cancer treatments, and DME items legitimately cost thousands to tens of thousands per service.
- IQR is computed globally across all service types. A $5,000 allowed amount is an outlier for a primary care E&M visit but completely normal for hip replacement surgery.
- The 3× multiplier is arbitrary and not validated.
- Removing outliers before feature engineering means the gold data has truncated distributions.

**Fix:**
1. Compute IQR within each `hcpcs_bucket` or `specialty × bucket` group, not globally
2. Use Winsorization (cap at P1/P99) instead of removal to preserve record count
3. Apply `log1p` transform before outlier detection (the target is already log-transformed for modeling)
4. Flag outliers with a binary indicator rather than removing them, and let the model decide

### No Data Drift Detection

The pipeline processes 11 years of data (2013-2023) without checking for structural changes:

- **ICD-10 transition (October 2015):** Changed diagnosis coding, potentially affecting HCPCS code distributions and risk scores
- **MACRA/MIPS implementation (2017+):** Merit-based payment adjustments changed the relationship between fee schedule and actual payments
- **COVID-19 (2020-2021):** Telehealth expansion created new service patterns, temporary payment increases, utilization drops
- **Sequestration changes:** 2% sequester was suspended in 2020-2021 and phased back in 2022

**Fix:**
1. Add year-over-year distribution comparisons in EDA: plot mean allowed amount by year × bucket to visually identify structural breaks
2. Compute Population Stability Index (PSI) between consecutive years for each feature
3. Consider training separate models for pre-COVID (2013-2019) and post-COVID (2020-2023) periods, or add explicit COVID indicator features
4. Weight recent years more heavily in training (exponential decay weighting by year)

### No Data Validation Framework

The pipeline has no formal schema validation between layers:

- Bronze → Silver: columns are renamed and typed, but no assertion that expected columns exist
- Silver → Gold: feature engineering assumes specific input columns without validation
- Gold → Model: models assume specific feature names and types

**Fix:** Add Pandera or Great Expectations schema definitions at each pipeline boundary. Even simple assertions (`assert "Avg_Mdcr_Alowd_Amt" in df.columns`) at the start of each script would catch data issues earlier.

## 6. Web App

### Pre-computed Only — No Live Inference

The app serves pre-aggregated statistics from Supabase. Users can only query combinations that exist in the pre-computed grid. If a specialty × bucket × state × POS combination was filtered out (n < 5 records), the user gets no result.

**Current gaps:**
- No custom submitted charge input (which is the #1 predictor in Stage 1)
- No continuous input for risk score, service counts, etc. — these are averaged in the pre-computation
- Cannot estimate cost for a new combination not seen in training data

**Fix:**
1. **Vercel Python serverless functions:** Deploy the RF or XGBoost model as a `/api/predict` endpoint. The XGBoost JSON model is ~50MB, within Vercel's 250MB limit. Users could input all 10 features and get real-time predictions.
2. **ONNX Runtime in WebAssembly:** Convert the XGBoost model to ONNX format, load it in the browser via `onnxruntime-web`. No server needed, predictions run client-side.
3. **Hybrid approach:** Keep pre-computed for common lookups (instant), fall back to live inference for custom inputs.

### No Confidence Indicators on Stage 1

The Stage 1 results show P10/P90 and record count, but the visual weight given to the mean allowed amount doesn't scale with confidence:

- A result based on 50,000 records and one based on 6 records look identical
- No warning when the estimate is from a small sample
- No indication when a fallback query was used (e.g., POS was dropped because exact match was empty)

**Fix:**
1. Add a confidence badge: "High confidence (10K+ records)" / "Moderate (100-10K)" / "Low (<100)"
2. Show a warning banner when displaying fallback results: "Exact match not found — showing results for all places of service"
3. Visually narrow the P10-P90 bar when sample size is small, widen when large

### Forecast Chart Missing Historical Data

The Forecast Explorer shows a single historical anchor point (last known year/value) and 3 forecast years. The full historical series (2013-2023) is available in `sequences.parquet` but not uploaded to Supabase.

**Fix:**
1. Create a `lstm_historical` table in Supabase with group keys + year + mean_allowed (from sequences.parquet's target_seq)
2. In the forecast chart, plot 2013-2023 as a solid line and 2024-2026 as a dashed line with P10-P90 shading
3. This would make the forecast context much richer — users can see the trend before evaluating the projection

### No Geographic Visualization

The Data Explorer has a state summary table but no map. A choropleth map showing mean allowed amounts by state would be:

- Immediately visually engaging (the most obvious feature of Medicare data is geographic variation)
- A strong portfolio differentiator
- Useful for identifying high/low cost states at a glance

**Fix:**
1. Use `react-simple-maps` or `d3-geo` with a US state GeoJSON
2. Color-code states by mean allowed amount (the data is already in `state_summary`)
3. Add click-to-filter: clicking a state could populate the Cost Estimator

### Mobile Responsiveness Gaps

The app has responsive layout (MUI Grid + breakpoints), but specific issues:

- Cost Estimator: the two-column form + results layout stacks vertically on mobile, but the toggle buttons and autocomplete fields may be too wide
- Forecast chart: Recharts responsive container works, but axis labels may overlap on small screens
- Model Comparison: the metrics table may need horizontal scrolling on mobile

**Fix:** Test at 375px width (iPhone SE) and 768px (iPad). Add `sx={{ overflowX: 'auto' }}` on table containers. Consider hiding less important columns on mobile.

### SEO and Performance

- **No Open Graph meta tags:** Sharing the URL on LinkedIn/Twitter shows generic text. Add OG image, title, description.
- **No page-level metadata:** Each route should have its own `<title>` and `description` via Next.js `generateMetadata`.
- **Static image optimization:** The EDA plots are served as raw PNGs (41-231KB each). Next.js `<Image>` component handles optimization, but the source images could be compressed further.
- **No analytics:** No way to know if anyone visits the app or which pages are popular. Add Vercel Analytics (free tier) or a simple Plausible/Umami script.

### Accessibility

- **No ARIA labels on interactive charts:** Recharts components don't have screen reader descriptions
- **Color contrast:** The P10/P50/P90 color differences (secondary.light / secondary.main / secondary.dark) may not have sufficient contrast
- **Keyboard navigation:** MUI components are generally keyboard-accessible, but the Recharts tooltips are mouse-only
- **No skip-to-content link:** Standard accessibility feature for keyboard users
