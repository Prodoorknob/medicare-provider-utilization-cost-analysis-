AllowanceMap Design System
Source of truth for colors, typography, layout, components, and interaction patterns for the AllowanceMap web application — a Medicare cost estimation and forecasting tool built on Next.js 16 + MUI v9 + Supabase. Covers both design intent and MUI implementation tokens.
STACK: Next.js 16 (App Router) · MUI v9.0.0 · Recharts 3.8 · Supabase · TypeScript 5.
THEME: Light-only. No dark mode. Healthcare legibility is the priority.
AUDIENCE: Medicare beneficiaries, caregivers, policy researchers, healthcare analysts.
01
Identity & Principles
AllowanceMap is a public-facing cost transparency tool for Medicare services. It must earn trust in the first ten seconds — the estimator is the first thing a user sees and interacts with. Every design decision serves that goal.
Brand Identity
Product: AllowanceMap
Tagline: Medicare cost transparency, built on data.
Primary Brand Color: Warm Navy (#1A4F8A)
Data Coverage: 2013–2026, 103M+ records, 63 state/territory markets
Voice: Plain English. Confident but never clinical. Explain what the numbers mean, not just what they are. Acknowledge uncertainty (P10/P90 bounds make this explicit). Never sensationalize healthcare costs.
Not: Government-document dense, startup-flashy, jargon-heavy.
Design Principles
1. Estimator first.
The cost estimator is the product. It must appear on the homepage without any scrolling past a marketing hero. A minimal identity header (name + one-line description) precedes the form, then immediately the tool. No testimonials, no feature lists, no marketing above the fold.
2. Trust through transparency.
Show confidence intervals (P10/P50/P90), never just a point estimate. Label synthetic vs. real data clearly. Disclaimers must be visible without being alarming — amber info alerts, not red warnings. State "based on 103M CMS records, 2013–2023" where space allows.
3. Warmth without clutter.
Healthcare is stressful. The UI should feel calm and organized. Use warm white backgrounds, generous whitespace, and soft shadows. No gradients on backgrounds, no animated heroes. Color is reserved for data meaning, not decoration.
4. Dense where it belongs.
The About page serves analysts and researchers who want methodology depth. That page can be information-dense (tabbed sections, in-page anchors, model metrics tables). The estimator and forecast pages must stay clean and task-focused.
5. Stage-aware color semantics.
Navy blue (#1A4F8A) = Medicare allowed amount (Stage 1 — what Medicare pays the provider).
Teal-green (#15755D) = Patient out-of-pocket cost (Stage 2 — what the beneficiary pays). This distinction is applied consistently across all result cards, charts, and labels throughout the app.
6. Numbers as heroes.
Dollar amounts in result cards use IBM Plex Mono and are significantly larger than surrounding copy — these are the answer the user came for. Everything else (labels, descriptions, footnotes) recedes visually. Never bury the result in paragraph text.

DO: Use navy (#1A4F8A) for anything related to what Medicare pays (allowed amount, forecast lines, Stage 1 results). Use teal (#15755D) for patient-facing costs (OOP, P10/P50/P90). Use amber (#B8763A) sparingly — disclaimers and info alerts only.

DON'T: Use primary blue on OOP results or teal on allowed amounts — the semantic distinction is load-bearing for user comprehension. Don't use color purely decoratively on backgrounds, borders, or icons. Don't add a third brand color for sections or "variety."

NOTE: The warm amber is intentionally restrained. If used for more than 2 elements per page it loses its "pay attention" meaning.

DON'T: Mix fonts within a label. Never use mono for navigation or button text — buttons must be Inter at all weights. Don't use serif or display fonts for the "healthcare feel" — Inter achieves warmth through weight and spacing, not typeface.

08
Interaction & Motion
Minimal animation. Healthcare users are often stressed — unprompted motion increases cognitive load. Transitions exist only to communicate state changes. No decorative animations, no loading skeletons that "shimmer," no parallax.
Transition Rules
Interaction Duration Easing
Button hover (bg color) 150ms ease-out
Input focus ring 150ms ease-out
Result card entrance 300ms ease-out
Tab panel transition 200ms ease-in-out
Loading spinner continuous linear
Result card entrance: When Stage 1 results arrive, the result card fades in with opacity: 0 → 1 and translateY: 8px → 0 over 300ms. Stage 2 results entrance after a short delay (150ms after Stage 1). No spring physics.

Loading state: Replace the "Estimate Costs" button label with a MUI CircularProgress (size=20, color="inherit") while both stages load in parallel. Do not disable the entire form — only the submit button. Show partial results if Stage 1 resolves before Stage 2.

Specialty chart: Chart data transitions with Recharts' built-in animation (isAnimationActive=true, animationDuration=500). No custom animation needed.

AVOID: Skeleton loaders that shimmer. Entrance animations on page load (page components mount without animation). Hover-triggered tooltip delays longer than 200ms.
09
Accessibility Standards
WCAG 2.1 AA minimum. Medicare users include older adults and people with vision impairments. High-contrast text, keyboard navigation, and screen-reader labels are non-negotiable.
Contrast Ratios
Pair Ratio Grade
#1C1917 on #FAFAF8 (body text) 16.8:1 AAA
#1A4F8A on #FAFAF8 (primary headings) 7.2:1 AAA
#15755D on #FAFAF8 (secondary headings) 5.9:1 AA
#FFFFFF on #1A4F8A (button text) 7.2:1 AAA
#57534E on #FFFFFF (secondary text) 6.1:1 AA
REQUIRED:
· All Autocomplete fields: aria-label describing the selection purpose
· Result cards: role="region" with aria-label="Stage 1 Medicare estimate"
· Forecast chart: aria-label on the chart container, accessible table fallback
· Loading state: aria-busy="true" on the results panel
· Error states: role="alert" on error messages
· Toggle buttons: aria-pressed on each ToggleButton
· Tab panels: aria-labelledby linking to the tab that controls them

FONT SIZE FLOOR: No text below 11px in the live UI. (Data labels on charts: 11px minimum.) The current app uses 9px swatch labels — acceptable only in this design doc, not in production.

TOUCH TARGETS: All interactive elements ≥ 44×44px. The ToggleButton group must have min-height 44px on mobile.
